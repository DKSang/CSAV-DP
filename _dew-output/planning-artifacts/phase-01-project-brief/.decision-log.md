# Phase 01 — Project Brief Decision Log

## Step 1 — Project Idea

**Question:** What data engineering project do you want to build?

**Decision:** A. Climate-smart agriculture / weather-agriculture advisory data product.

**Status:** Confirmed by user.

**Date:** 2026-05-26

## Step 2 — Target User / Consumer

**Question:** Who will use the data product in the MVP?

**Decision:** A and C.

- **A. Data analyst** — primary technical consumer. Needs clean, documented, analysis-ready data products such as Gold marts, KPIs, and quality evidence.
- **C. Farmer / end user / customer** — primary domain/end consumer. Needs simplified advisory or risk signals, likely served through a later web app or user-facing feature.

**Interpretation for MVP:** The MVP should first make the data reliable and useful for analysts, while keeping farmer-facing advisory use cases as an explicit design constraint. This avoids jumping too early into UX or production advisory claims before the data foundation is validated.

**Status:** Confirmed by user.

**Date:** 2026-05-26

## Step 3 — Decision / Action / Question Supported

**Question:** What decision, action, or question should the data product support?

**Decision:** A and D.

- **A. Monitor risk** — the MVP should help users monitor climate/agricultural risk by location and time.
- **D. Recommend an action** — the product should move toward advisory-style action recommendations for farmers or decision-makers.

**Interpretation for MVP:** Risk monitoring is the primary decision-support capability. Action recommendation is included as an advisory direction, but must be treated as a hypothesis until source quality, rules, domain thresholds, and limitations are validated. The project must avoid presenting recommendations as production-grade advice in the MVP.

**Status:** Confirmed by user.

**Date:** 2026-05-26

## Step 4 — Expected Data Product Type

**Question:** What is the expected MVP data product type?

**Decision:** B, A, and E.

- **B. Gold data mart** — core data product. This is the foundation for analyst use, KPI/risk modeling, quality checks, and traceability.
- **A. Dashboard** — analyst-facing serving layer for monitoring, comparison, and explanation of risk signals.
- **E. Web app feature** — farmer/end-user-facing serving layer for simplified risk/advisory information.

**Interpretation for MVP:** The implementation should prioritize the Gold data mart first, then dashboard, then web app feature. This prevents the project from jumping into UI before validated data, grain, KPIs, and risk/advisory logic exist.

**Status:** Confirmed by user.

**Date:** 2026-05-26

## Step 5 — Project Type Gate

**Question:** What project type should govern quality, scope, and evidence expectations?

**Decision:** B. Portfolio project.

**Interpretation for MVP:** The project should be strong enough to demonstrate end-to-end data engineering capability, including clear business purpose, source evidence, Bronze/Silver/Gold modeling, data quality checks, documentation, and a visible serving layer. It does not need production-grade SLAs, full monitoring, or guaranteed advisory reliability yet.

**Status:** Confirmed by user.
